# Launch Checklist

Use this checklist to share BountyOps in a way that earns real interest without spam.

## Repo Readiness

- Keep the live demo link at the top of the README.
- Keep the repository description specific: reasoning-agent demo for GitHub bounty monitoring, CI checks, payout-risk triage, and no-spam claim protection.
- Keep topics focused on real discovery terms: `ai-agents`, `github-automation`, `workflow-automation`, `developer-tools`, `ci-monitoring`, `open-source-bounties`, `gmail-automation`, `mcp`.
- Make the first review path obvious: live demo, architecture, workflow template, use cases, public data boundary.
- Keep examples public-safe: no Gmail bodies, cookies, tokens, bank data, private claim URLs, or credentials.

## Manual Sharing Path

1. Add the repo to the GitHub profile featured section and keep it pinned.
2. Use the short post from [SHARE_KIT.md](SHARE_KIT.md) in one relevant community at a time.
3. Share only where practical agent, GitHub automation, CI monitoring, bounty, or developer-tool demos are welcome.
4. Ask for feedback on the workflow pattern, not generic stars.
5. Wait for replies before posting follow-ups.

## Good Follow-Up Angles

- "Could this action policy reduce noisy agent comments in your workflow?"
- "What would you add to make this useful for release readiness or support triage?"
- "Would a CLI/template version be more useful than the static demo?"
- "Which state signals should be first-class in GitHub operations agents?"

## Do Not Do

- Do not buy, trade, or request artificial stars.
- Do not mass-DM maintainers or recruiters.
- Do not post the same message across unrelated issues.
- Do not imply the demo includes private payout or Gmail data.
- Do not overstate production adoption; present it as a public-safe reference and reusable pattern.

## Weekly Maintenance

- Refresh the README if the demo, architecture, or workflow template changes.
- Keep open issues at zero unless there is a real roadmap item.
- Add concrete screenshots or short demo clips when available.
- Convert repeated feedback into issues or docs updates.
- Track star count and traffic as feedback signals, not as a reason to spam.
